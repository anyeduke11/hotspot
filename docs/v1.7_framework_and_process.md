# Hotspot v1.7 新框架与业务流程设计

> **版本**: v1.7  
> **基线**: v1.6（CodeGarden Phase 2b 完成后的代码基线，详见 [CODE_WIKI.md](./CODE_WIKI.md)）  
> **对应 PRD**: [docs/hotspot_v1.7_PRD.md](./hotspot_v1.7_PRD.md)  
> **核心目标**: 从"信息平台"升级为"主动认知协作者"，补齐 **信息 → 知识 → 行动** 链路上的 6 个断裂环节。  

---

## 1. 设计目标与第一性原理

v1.6 已具备热点聚合、质量门禁、知识库、CodeGarden 项目/服务/资源/编排等能力，但用户仍需在"筛选、提取、关联、评估、行动、内化"六个环节做大量手动决策。v1.7 以"减少认知摩擦"为第一原则，通过以下方式实现升级：

| 断裂环节 | v1.6 状态 | v1.7 目标 | 优先级 |
|---|---|---|---|
| 筛选 (Signal Filtering) | 互斥分类 + 基础搜索 | 多维标签 + AND/OR 交叉筛选 + 个性化排序 | P0 |
| 提取 (Understanding) | 完全手动阅读 | 自动提取标签 / 概念 / 技术栈 / CVE | P0 |
| 关联 (Contextualization) | 孤岛式存储 | 跨层搜索 + 上下文推荐 + 项目桥接 | P1 |
| 评估 (Decision) | 无主动告警 | 规则引擎 + SSE PUSH + 技术栈影响分析 | P1 |
| 行动 (Action) | 仅 Todo 记录 | 收藏/告警 → 知识条目 / Todo / CodeGarden 事件自动桥接 | P1 |
| 内化 (Internalization) | 读完即忘 | SM-2 间隔重复 + 笔记空间 | P2 |

**一句话定位**：让系统从"被动信息仓库"变成"主动认知协作者"。

## 2. v1.7 总体架构

### 2.1 架构核心：Hotspot ↔ Agent 双向环

v1.6 是线性管道：

```
Source → Collector → Hotspot DB → User → (手动) → Knowledge(.md)
```

v1.7 是双向环：

```
┌──────────────────────────────────────────────────────────────────┐
│                        Hotspot 平台                               │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────────────┐  │
│  │   采集层      │  │   知识引擎    │  │   CodeGarden / Todo    │  │
│  │ 8 Collectors │  │ 提取/标签/推荐 │  │ 项目/服务/资源/编排     │  │
│  │ Cubox/Bookmark│  │ SM-2/告警/搜索 │  │                        │  │
│  └──────┬───────┘  └──────┬───────┘  └───────────┬────────────┘  │
│         │                  │                      │               │
│  ┌──────▼──────────────────▼──────────────────────▼──────┐       │
│  │           SQLite 主存储 (37+ 表，WAL)                  │       │
│  │  hotspots / knowledge_items / todos / cg_* / ...      │       │
│  │  + kv_cache 缓存层                                      │       │
│  └──────┬────────────────────────────────────────────────┘       │
│         │                                                          │
│  ┌──────▼──────────────┐  ┌─────────────────────────────┐        │
│  │   REST API / SSE    │  │   Task Queue (FS)           │        │
│  │   /api/*            │  │   knowledge/learning/tasks/ │        │
│  └──────┬──────────────┘  └──────────────┬──────────────┘        │
└─────────┼────────────────────────────────┼───────────────────────┘
          │ HTTP API / FSWatch             │ 文件系统
          │                                │
┌─────────▼────────────────────────────────▼───────────────────────┐
│                         Agent (CLI)                               │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐  │
│  │ Task Queue Watcher│  │ Skill Executor  │  │ Hotspot API Client│  │
│  │ (phase-locked)   │  │ (LLM/Regex/Script)│  │ (写回结果)        │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ CLI 工具层：hotspot-agent {start|stop|status|tasks|skill|cubox|bookmark|knowledge} │
│  └─────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────┘
```

**关键变化**：
- Hotspot 生产任务（extract / analyze / link / generate）写入 `tasks/pending/`，Agent 消费并写回 API。
- Agent 生产的新知识条目（`.md`）触发 Hotspot 的 FSWatch，进一步产生新的任务或生命周期推进。
- Scheduler 以 APScheduler 为核心，连接两个方向的数据流。

### 2.2 双向环数据流

**Hotspot → Agent 方向**：

```
新文章入库 / 收藏文章
    → 写入 hotspots / favorites 表
    → lifecycle = signal
    → agent_task_consumer job 创建 task-{id}.md 到 tasks/pending/
    → Agent 轮询读取任务
    → 执行对应 Skill
    → POST /api/agent/knowledge 写回提取结果
    → Hotspot 更新 lifecycle 并触发下一轮任务
```

**Agent → Hotspot 方向**：

```
Agent 执行 Skill 获得新知识
    → POST /api/agent/knowledge
    → 写入 knowledge/items/{id}.md + 更新 knowledge_items 表
    → 更新 kv_cache
    → 触发概念关联、自动提取、推荐侧栏刷新
    → 用户在前端 / Obsidian 消费
    → 用户行为（收藏/笔记/评分）回到 Hotspot
    → Hotspot 生成新的 pending task
```

## 4. SAG 生命周期状态机

v1.7 用五阶段状态机替换 `compiled: bool`。

```
                   ┌─────────────┐
                   │   signal    │  ← 刚采集到 / 收藏后 / Agent 创建任务
                   └──────┬──────┘
                          │ 自动/手动标签提取完成
                   ┌──────▼──────┐
                   │amplify:     │
                   │ tagged      │  ← 标签/概念/技术栈已提取
                   └──────┬──────┘
                          │ 知识图谱关联完成
                   ┌──────▼──────┐
                   │amplify:     │
                   │ linked      │  ← 已关联 concept / related_items
                   └──────┬──────┘
                          │ 上下文信息完备
                   ┌──────▼──────┐
                   │amplify:     │
                   │ complete    │  ← 可生成知识条目
                   └──────┬──────┘
                          │ Agent / 用户生成 .md
                   ┌──────▼──────┐
                   │  generate   │  ← 已写入 knowledge/items/{id}.md
                   └─────────────┘
```

**状态转移规则**：

| 转移 | 触发条件 | 操作 |
|---|---|---|
| `signal` → `amplify:tagged` | 自动标签提取完成，或用户手动打标签 | 写入 `tasks/pending/` link 任务 |
| `amplify:tagged` → `amplify:linked` | 概念关联 / 知识图谱节点创建完成 | 写入 `tasks/pending/` generate 任务 |
| `amplify:linked` → `amplify:complete` | 关联信息完备，上下文建立 | 等待生成条件满足 |
| `amplify:complete` → `generate` | 知识条目已生成到 `knowledge/items/` | 通知用户 / 加入复习队列 |
| 任意状态可手动回退 | 用户修改后重新处理 | 重新创建对应类型的 pending task |

**数据迁移**：
- `compiled = 1` → `lifecycle = 'generate'`
- `compiled = 0` → `lifecycle = 'amplify:complete'`

## 6. OKF + LLM-Wiki 2.0 统一存储

### 6.1 核心原则

- `knowledge/` 目录下的 `.md` 文件是知识资产的**唯一源数据**（OKF）。
- SQLite（`knowledge_items`）是读缓存 + 查询加速。
- `kv_cache` 是热数据缓存，各端独立，不参与跨端同步。
- Obsidian 可直接读取 `knowledge/items/`，用户编辑后由 FSWatch 感知并回写 SQLite。

### 6.2 目录结构（保持不变）

```
knowledge/
├── _MAP.md              ← 知识地图（自动生成索引）
├── _SCHEMA.md           ← 数据模型合约
├── SOUL.md              ← 角色画像（Agent 自动生成）
├── items/               ← L1: 知识条目 (*.md)
│                           lifecycle / tags / tech_stack / concepts
├── concepts/            ← L2: 提取的概念
├── learning/            ← L3: 学习计划 + 任务队列
│   └── tasks/
│       ├── pending/     ← Hotspot 写入，Agent 读取
│       ├── processing/  ← Agent 正在处理
│       ├── done/        ← 已完成
│       └── failed/      ← 失败 + error.md
├── content/             ← L4: 内容创作
│   ├── drafts/
│   └── calendar.json
└── summaries/           ← 每周摘要
```

```
用户 / Agent 写入 .md
    → watchdog 感知
    → knowledge_sync.sync_item_to_db() 解析 frontmatter
    → upsert knowledge_items

```

## 7. KV 缓存层

### 7.1 表结构

```sql
CREATE TABLE kv_cache (
    key        TEXT PRIMARY KEY,
    value      TEXT NOT NULL,        -- JSON 序列化
    etag       TEXT,                 -- 文件内容 MD5 或版本号
    expires_at TEXT,                 -- ISO-8601 过期时间
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
CREATE INDEX idx_kv_cache_expires ON kv_cache(expires_at);
```

### 7.2 缓存键与过期策略

| 键类型 | 示例 | 过期时间 | 说明 |
|---|---|---|---|
| 单条 item | `item:{id}` | 60s | 缓存解析后的 knowledge item |
| 列表 | `items:list:{filter_hash}` | 30s | 列表查询结果 |
| 标签索引 | `items:by_tag:{tag}` | 30s | 按标签聚合 |
| FTS 索引 | `fts:index` | 5min | unified_fts 预计算结果 |
| 任务队列 | `agent:tasks:pending` | 10s | pending 任务摘要 |

### 7.3 失效策略

- 文件变化时更新 `etag`，API 写操作主动 `DELETE FROM kv_cache WHERE key LIKE '...%'`。
- 过期键在 `kv_cache_cleanup` job 中每 30min 清理。
- 命中率目标：`> 80%`。

## 8. Agent 协议与任务队列

### 8.1 任务文件格式

每个任务是一个 Markdown 文件，位于 `knowledge/learning/tasks/pending/`：

```yaml
---
task_id: "task-20260722-001"
type: "extract"               # extract | analyze | link | generate | review
status: "pending"
created_at: "2026-07-22T10:00:00Z"
target_type: "hotspot"        # hotspot | favorite | knowledge
target_id: "a1b2c3"
priority: 1                   # 1-5, 5 最高
source: "hotspot-api"         # hotspot-api | scheduler | user
params:
  model: "gpt-4o-mini"
  skill_id: "extract-tags"
---
```

### 8.2 任务类型

| type | 触发时机 | Agent 动作 | 写回结果 |
|---|---|---|---|
| `extract` | 新文章入库 | 提取标签/概念/技术栈 | POST /api/agent/knowledge |
| `analyze` | 收藏后 | 深度分析，生成摘要 | POST /api/agent/knowledge |
| `link` | 概念提取后 | 关联知识图谱 | POST /api/agent/knowledge |
| `generate` | 信息完备后 | 生成知识条目 .md | POST /api/agent/knowledge |
| `review` | 定时触发 | 评估知识质量 | POST /api/agent/tasks/{id}/complete |

### 8.3 Phase-locked 轮询协议

```
Agent 启动:
  1. GET /api/config → collect_interval_seconds
  2. poll_interval = collect_interval_seconds (默认 300s)
  3. 启动轮询循环

轮询循环:
  while True:
    1. GET /api/agent/tasks?status=pending&limit=10
    2. if tasks:
         for task in tasks:
           执行 Skill
           POST /api/agent/tasks/{id}/complete
    3. else:
         GET /api/hotspots?lifecycle=signal&limit=10
         if 有新热点:
           创建本地任务并处理
           POST /api/agent/knowledge
    4. sleep(poll_interval)
```

**动态调整**：任务多时间隔缩短（最小 60s），空轮询时间隔延长（最大 600s）。

### 8.4 Skill 配置

为避免与现有 `skills` 表（Phase 41 的 skill 市场）冲突，Agent Skill 配置使用 **`skill_configs`** 表：

```json
{
  "id": "extract-tags",
  "name": "标签提取器",
  "description": "从文章内容中提取标签、概念、技术栈",
  "type": "llm",
  "config": {
    "model": "gpt-4o-mini",
    "temperature": 0.1,
    "max_tokens": 1000,
    "prompt_template": "...",
    "output_format": { "tags": ["string"], "concepts": ["string"], "tech_stack": ["string"] }
  },
  "target_types": ["hotspot", "favorite"],
  "created_at": "2026-07-22T00:00:00Z"
}
```

Skill 类型：`llm` / `regex` / `hybrid` / `script`。

### 8.5 Agent API 端点

| Method | Path | 说明 |
|---|---|---|
| GET | `/api/agent/tasks` | Agent 轮询获取 pending 任务 |
| POST | `/api/agent/knowledge` | Agent 写回知识条目 |
| POST | `/api/agent/tasks/{id}/complete` | Agent 标记任务完成/失败 |

## 9. 用户旅程业务流程（M1-M12）

### 9.1 M1：简报与态势感知

**触发**：每日首次打开 dashboard / 离线归来。

**流程**：
1. `digest_generator` job 每日 08:00 基于昨日热点 + `reading_states` 生成简报，写入 `digests`。
2. `source_health_check` job 每 15min 检查各源采集量 vs 7 日基线。
3. 首页根据当前时间和 `digests` 未读状态自动进入简报模式或扫描模式。

**输出**：一句话摘要 + 3 篇关键文章 + 数据源健康指示器。

### 9.2 M2：多维标签与交叉筛选

**触发**：用户在首页使用标签筛选。

**流程**：
1. 后端维护 `tags` 层级表 + `hotspot_tags` 多对多表。
2. 列表接口支持 `tags=ai,security&tag_mode=and`（交叉）或 `tag_mode=or`（并集）。
3. `TagSelector` 组件提供多选 + AND/OR 切换。

**输出**：命中所有/任一标签的文章列表。

### 9.3 M3：自动知识提取

**触发**：新文章入库 / 收藏。

**流程**：
1. `extract_service` 按三层提取器处理：
   - `RegexExtractor`：CVE / CNVD（置信度 1.0）
   - `KeywordExtractor`：技术关键词（置信度 0.7）
   - `CategoryDomainExtractor`：分类域映射（置信度 0.5）
2. 结果写入 `hotspot_tags` 并更新 `hotspots.tags` JSON 冗余字段。
3. 触发 SAG lifecycle → `amplify:tagged`。

**输出**：文章附带自动标签、概念、技术栈。

### 9.4 M4：SM-2 间隔复习

**触发**：复习队列非空 / 每日 17:00。

**流程**：
1. `review_scheduler` job 每 6h 查询 `sm2_reviews.next_review_at`。
2. 用户进入 `/reviews`，对卡片自评 grade（0-5）。
3. `sm2_schedule(grade, easiness, interval, reps)` 更新下一次复习时间。

**输出**：个性化复习队列。

### 9.5 M5：技术栈桥接

**触发**：新文章提取到 `tech_stack` 标签。

**流程**：
1. `tech_stack` 表记录个人技术栈及关联的 `cg_projects`。
2. `extract_service` 提取 tech_stack 后匹配 `tech_stack` 表。
3. 命中项目则创建 `cg_events` + Todo（`source_article_id` 关联）。

**输出**：项目影响评估 + 待确认 Todo。

### 9.6 M6：规则告警系统

**触发**：新文章入库 / 手动触发。

**流程**：
1. `alert_evaluator` job 每 60s 扫描未评估新文章。
2. 按 `alert_rules.condition` JSON DSL 评估：
   ```json
   {
     "type": "tag_match",
     "operator": "AND",
     "conditions": [
       { "field": "tags", "op": "contains_any", "value": ["CVE", "vulnerability"] },
       { "field": "tags", "op": "contains_any", "value": ["fastapi", "langchain"] }
     ],
     "actions": { "push": ["sse"], "auto_mark": "urgent", "auto_todo": true }
   }
   ```
3. 命中后写入 `alerts`（或复用通知表），SSE 推送到前端，可选自动创建 Todo。

**输出**：红色告警横幅 + 告警中心 Inbox。

### 9.7 M7：统一跨层搜索

**触发**：用户在前端搜索框输入关键词。

**流程**：
1. `unified_fts` 虚拟表基于 `unified_search` 视图（hotspots / knowledge / todos / favorites / projects）。
2. `/api/search?q=...&sources=...` 返回聚合结果，按 source_type 分组。
3. 前端 `useSearch` hook 做 debounce（300ms）。

**输出**：同一页展示 5 层结果。

### 9.8 M8：上下文感知知识推荐

**触发**：用户打开深度阅读模式。

**流程**：
1. 取当前文章 `tags` + `concepts` + `tech_stack`。
2. 在 `knowledge_items` 和 `concepts` 中查找共享标签或相关概念。
3. 按 `personal_profile` 权重排序，过滤已读条目。

**输出**：右侧推荐侧栏显示相关知识条目。

### 9.9 M9：笔记空间

**触发**：用户在深度阅读模式点击"笔记"Tab。

**流程**：
1. `annotations` 表记录 `entity_type` + `entity_id` + `content`。
2. `NoteEditor` 组件提供简化 Markdown 编辑。
3. 笔记与阅读状态一起参与跨端同步。

**输出**：文章/概念旁的用户笔记。

### 9.10 M10：隐式个性化

**触发**：用户打开、停留、收藏、记笔记、忽略文章。

**流程**：
1. `reading_states` 记录 `opened_count`、`total_dwell_ms`、`last_opened_at`。
2. `profile_updater` job 每 30min 计算信号权重：
   - 打开 +0.1
   - 停留 >60s +0.2
   - 收藏 +0.5
   - 记笔记 +0.3
   - 忽略 -0.05
3. `weight = max(-2, min(2, weight_old * 0.95 + signal))`。
4. `profile_decay` job 每日 03:00 全局衰减 5%。

**输出**：分类/标签/源权重，影响推荐排序。

### 9.11 M11：数据源完整性仪表盘

**触发**：每 15min / 首页加载。

**流程**：
1. `source_health_check` job 比较近 N 小时采集量与 7 日基线。
2. 偏差在 ±30% 内为 green，±30%~60% 为 yellow，>60% 或连续 0 条为 red。
3. `SourceHealthBar` / `SourceHealthIndicator` 展示状态。

**输出**：数据源健康指示器。

### 9.12 M12：收藏 → 知识提升

**触发**：用户收藏文章。

**流程**：
1. `POST /api/favorites` 写入 `favorites`。
2. 同时创建 `knowledge/items/{id}.md`（仅 frontmatter + 原文链接），`lifecycle = signal`。
3. `agent_task_consumer` 创建 extract 任务。
4. Agent 提取后更新 lifecycle → `amplify:tagged`。
5. 用户可在 Obsidian 中直接阅读和编辑。

**输出**：收藏自动进入知识生命周期。

## 15. 风险与对策

| # | 风险 | 概率 | 影响 | 对策 |
|---|---|---|---|---|
| 1 | FTS5 统一搜索性能不达标 | 中 | 高 | fallback 到各层独立搜索 + 前端合并 |
| 2 | 自动标签提取准确率低 | 中 | 中 | 置信度阈值 + 用户确认 |
| 3 | SM-2 复习队列冷启动 | 高 | 低 | 默认每天推送 3 条 |
| 4 | 隐式学习偏差 | 中 | 低 | weight 衰减上限 5%/day + 手动重置 |
| 5 | 告警规则太泛导致疲劳 | 中 | 中 | 默认空规则 + cooldown |
| 6 | 多表 JOIN 性能下降 | 低 | 高 | `tags` JSON 冗余字段 |
| 7 | 新增 10 个 scheduler job 影响采集 | 低 | 中 | ThreadPoolExecutor 隔离 |
| 8 | Agent 轮询延迟导致任务堆积 | 中 | 中 | 动态调整轮询间隔 + 并发处理 |
| 9 | Agent 与 Hotspot 状态不一致 | 中 | 中 | 幂等性校验 + 重试机制 |
| 10 | kv_cache 与文件系统不一致 | 中 | 低 | etag 校验 + 自动失效 |
