## 1. 测试策略总览

### 1.1 测试金字塔

```
        /\
       /  \
   E2E /    \   ~8 tests (关键用户旅程)
      /______\
        /\
       /  \
Integration /    \   ~35 tests (跨服务/跨模块)
      /______\
        /\
       /  \
  Unit /    \   ~90 tests (服务/仓库/工具函数)
      /______\
```

### 1.2 测试分层定义

| 层级 | 范围 | 运行速度 | 是否阻塞 CI |
|------|------|----------|------------|
| **Unit** | 单个函数 / Repository / Service，无 DB 或用内存 SQLite | < 1s / 文件 | 是 |
| **Integration** | API + DB + 文件系统，使用 TestClient + tmp_path | < 5s / 文件 | 是 |
| **E2E** | 完整用户旅程：采集 → Agent → 知识库 → 前端渲染 | < 60s / 文件 | 是 |
| **Frontend Unit** | React 组件 + Hook，msw 拦截 API | < 1s / 文件 | 是 |
| **Performance** | 搜索/提取/告警 P95 延迟 | 手动 / 每日 | 否（但门禁） |

### 1.3 测试原则

1. **PRD 章节全覆盖**: 每个 PRD 章节至少对应一个测试文件或验收用例。
2. **TDD**: 开发计划中每个 Task 的 Step 1 必须是失败测试。
3. **测试即文档**: 测试用例名称直接描述业务行为，例如 `test_tag_and_mode_filters_hotspots`。
4. **无外部依赖**: 所有测试在本地可运行，不依赖 LLM / 外部 API / 真实网络。
5. **幂等 & 隔离**: 每个测试使用独立数据库（`tmp_path` + monkeypatch）或内存存储。
6. **失败即中断**: CI 中任何一层失败即阻断合并。

## 3. 单元测试规范

### 3.1 测试文件命名

- Python: `backend/tests/test_<module_name>.py`
- TypeScript: `frontend/src/components/<Component>.test.tsx` 或 `frontend/src/hooks/use<Name>.test.ts`

### 3.2 数据库隔离

所有后端单元/集成测试必须使用以下模式:

```python
import pytest
from backend.repository.db import init_db, get_connection

@pytest.fixture
def db(tmp_path, monkeypatch):
    db_path = tmp_path / "test.db"
    monkeypatch.setattr("backend.repository.db.DB_PATH", db_path)
    monkeypatch.setattr("backend.config.config.db_path", db_path)
    init_db()
    yield get_connection()
```
